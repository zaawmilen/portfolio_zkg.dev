04 — Engineering Approach

How I build systems

The decisions below aren't principles copied from a blog. They're trade-offs I've reasoned through in real systems — where the wrong call meant a race condition in production, a user losing a bid they actually won, or a deployment that required a 2am rollback.

01 — Failure Handling
Three-Layer Concurrency on Bids

Two users submit bids at the same millisecond across two server instances behind a load balancer. SELECT FOR UPDATE alone isn't enough — in pgBouncer transaction pooling mode, session-scoped advisory locks release when the connection returns to the pool, not when the transaction commits.

Fix: three layers working together — SELECT FOR UPDATE prevents dirty reads, pg_advisory_xact_lock (transaction-scoped) serializes finalization across instances, and a CAS gate (UPDATE WHERE current_price < bid_amount) is the final atomic commit guard.

Every bid attempt written to append-only bid_events — accepted and rejected — inside the same transaction
Idempotency & Safe Retries

A client submits a bid, the network drops before the response arrives. The user retries. Without idempotency, two bids are recorded — or worse, two charges.

Fix: clients pass an x-idempotency-key header. The server claims the key atomically with INSERT ... ON CONFLICT DO NOTHING before the transaction starts. A duplicate key in pending state returns 409 Conflict.

All mutation endpoints return deterministic, client-actionable error shapes — never raw stack traces
Graceful Shutdown & Zero-Drop Deploys

On Fly.io, deployments send SIGTERM before starting the new version. Without graceful shutdown, in-flight bid transactions are killed mid-execution — the client gets a connection reset and retries, potentially duplicating the bid.

Fix: SIGTERM stops accepting new connections immediately, then tracks and destroys open keep-alive sockets. BullMQ workers drain their active job. DB pool and Redis close cleanly.

Zero dropped requests during rolling deploys — verified against live Fly.io environment
02 — AI & Search Pipeline
Hybrid Search — pgvector + RRF

"Flood damaged toyota with intact frame" doesn't match a listing titled "2020 Toyota Tacoma TRD Off-Road" by keyword overlap alone.

Fix: two signals fused via Reciprocal Rank Fusion. pgvector cosine similarity on OpenAI text-embedding-3-small vectors handles semantic intent. PostgreSQL tsvector handles exact keyword matches.

Rejected: Elasticsearch (extra infra) and weighted score fusion (incomparable scales)
Async Embedding Pipeline

Generating embeddings synchronously couples HTTP response latency to OpenAI API latency — 200–500ms per call, with failure propagation if the API is unavailable.

Fix: embeddings generated fire-and-forget via a dedicated BullMQ worker. The auction creation response returns immediately after the DB write. Falls back to keyword-only search if the embedding is not yet generated.

safeQueueAdd() wraps every enqueue — 3s timeout, fail-open, never blocks HTTP response
Streaming RAG Analysis

Static auction details don't help a bidder decide whether a price is fair. Waiting for a full AI response before rendering degrades perceived performance.

Fix: RAG pipeline retrieves similar auctions as context, then streams an Anthropic Claude analysis via Server-Sent Events. First token in ~300ms, full analysis in 5–8 seconds.

SSE chosen over WebSocket — unidirectional stream, no client-to-server messaging needed
03 — Technology Trade-offs
Cursor vs. OFFSET Pagination

OFFSET 10000 requires Postgres to scan and discard 10,000 rows. At Copart scale — millions of listings — page 500 takes seconds. The problem compounds as inventory grows.

Fix: keyset pagination on (endTime DESC, id ASC). Page 1 and page 10,000 have identical query plans — consistent latency regardless of depth.

OFFSET kept as fallback for backward compatibility — cursor takes precedence when present
BullMQ vs. Database Queue

Three operations — embedding generation, auction expiry, outbid notifications — are too slow or risky to run synchronously on the HTTP path.

BullMQ chosen over pg-boss: Redis is already in the stack, BullMQ has better tooling (Bull Board dashboard), and delayed jobs are a first-class feature. Trade-off acknowledged: pg-boss offers transactional enqueue.

jobId: expire-${'{auctionId}'} deduplicates expiry jobs — safe across retries and rolling deploys
Why PostgreSQL, not MongoDB

Bid data is financial-like — every write must be correct or not happen at all. MongoDB's default write model doesn't give cross-collection transaction guarantees without explicit sessions.

PostgreSQL's ACID guarantees, row-level locking, and pgvector extension cover both the transactional requirements and the AI search pipeline in a single data store.

At 3M+ rows: HNSW index on embedding column replaces flat cosine scan — O(log n) vs O(n)
04 — Production Signals
Append-Only Audit Log

Application logs are ephemeral. When a user disputes a bid outcome, parsing log files to reconstruct what happened is slow and unreliable.

Fix: every bid attempt is written to an immutable bid_events table. Accepted bids write inside the transaction. Rejections write outside — always persisted, even on rollback.

No FK constraint on bid_id — audit log is independent of bids table lifecycle
Correlation IDs & Error Fingerprinting

A user reports an error at 14:32. Without request-level tracing, finding the exact log entry means scanning thousands of lines across multiple services.

Fix: every request is tagged with a correlation ID. The ID propagates into pino logs, error responses, and the bid_events audit table. Unhandled errors are tagged with a fingerprint for log aggregator grouping.

correlationId in 500 responses — support can cross-reference logs without codebase access
Dual-Layer Rate Limiting

A single per-auction rate limit can be bypassed by spreading bot activity across many auctions simultaneously — staying under the per-auction threshold while exceeding any reasonable human bid rate globally.

Fix: two independent Redis sliding windows as a Lua script (atomic). Per-auction: 10 bids/min. Global: 30 bids/min. Global check fires first.

Fail-open on Redis error — availability over security for rate limiting; DB locking still guarantees correctness
