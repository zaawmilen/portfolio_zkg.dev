04 — Engineering Approach

How I build systems

The decisions below aren't principles copied from a blog. They're trade-offs I've reasoned through in real systems — where the wrong call meant a race condition in production, a user losing a bid they actually won, or a deployment that required a 2am rollback.

01 — Failure Handling
Race Conditions on Concurrent Bids

Two users submit bids at the same millisecond. Without locking, both reads see the same current highest bid — both pass validation — and both commit. The auction now has two "winning" bids.

Fix: SELECT FOR UPDATE locks the auction row for the duration of the transaction. The second writer blocks until the first commits, then reads the updated state and correctly loses.

Considered: optimistic locking (version field) — rejected because bid windows are short and retry UX is worse than a queue
Retries & Duplicate Submissions

A client submits a bid, the network drops before the response arrives. The user retries. Without idempotency, two bids are recorded — or worse, two charges.

Fix: unique DB constraint on (user_id, auction_id, amount) makes retries naturally safe. A duplicate hits the constraint and returns a structured 409 Conflict — not a 500 — so the client knows the bid already landed and doesn't retry again.

All mutation endpoints return deterministic, client-actionable error shapes — never raw stack traces
Dropped WebSocket Connections

A user submits a bid 50ms before their connection drops. Did it land? Replaying missed WebSocket events is brittle — events may have been lost, reordered, or partially processed server-side.

Fix: server-authoritative state on reconnect. The client re-fetches full current auction state from REST on reconnect — including whether their last bid succeeded. No event replay. No client-side state guessing.

Trade-off: heavier reconnect payload vs. guaranteed consistency — correct choice for financial-like data
02 — Technology Trade-offs
Why WebSockets, not SSE or Polling

Long-polling creates a new HTTP request per update — too much overhead for sub-second bid events. Server-Sent Events are unidirectional — the server pushes, but the client can't send bids back over the same channel.

WebSockets give a persistent, bidirectional connection — the server pushes new bid events to all clients, and each client sends bids back on the same socket. One connection, both directions, low latency.

Cost: connection state management on the server — worth it for real-time bidding UX
Why PostgreSQL, not MongoDB

Bid data is financial-like — every write must be correct or not happen at all. MongoDB's default write model (document-level atomicity) doesn't give cross-collection transaction guarantees without explicit sessions, which adds complexity for no benefit here.

PostgreSQL's ACID guarantees, row-level locking, and serialisable isolation are exactly what bid integrity requires. The relational model also fits — auctions, items, users, and bids have clear foreign-key relationships.

MongoDB used for document-heavy data (profiles, logs) where schema flexibility matters more
Pessimistic vs. Optimistic Locking

Optimistic locking (version field + retry-on-conflict) works well when conflicts are rare and retrying is cheap — e.g. updating a user profile.

For live auctions, conflicts are expected and frequent. A failed bid shouldn't silently retry with a new amount — the user needs an explicit "you were outbid" response. Pessimistic locking blocks concurrent writers and surfaces the conflict immediately, giving the client a definitive, actionable answer.

Rule: optimistic for low-contention data, pessimistic when correctness on first write matters
03 — Production Signals
Testing Strategy

Unit tests cover pure business logic — bid validation rules, price calculation, permission checks — in isolation, with mocked DB layers. Fast, deterministic, run on every commit.

Integration tests hit a real test database and cover the full request lifecycle — including transaction rollback on constraint violations. These catch the bugs unit tests structurally can't.

API contracts tested with Postman collections — exported and version-controlled alongside code
Deployment & Rollback

CI/CD pipeline via GitHub Actions: lint → test → build → deploy. Staging environment mirrors production — no untested code reaches production. Merging to main triggers a deploy only after all checks pass.

Database migrations run separately from code deploys, and are written to be backwards-compatible — new columns are nullable by default so a rollback doesn't break the previous build.

Rollback strategy defined before every deploy — not figured out during an incident
Observability & Error Visibility

Every API error is logged with structured context — user ID, route, error type, timestamp — so a production issue can be traced to the exact request without reading raw stack traces. Sensitive data is never logged.

WebSocket disconnections and reconnection events are tracked. If a user disconnects mid-auction, the server cleans up their session state and broadcasts the updated active-user count — the system degrades gracefully, not silently.

Health check endpoint on all services — infrastructure can self-heal without manual intervention
